## 摘要

本文档系统性地记录了基于 OpenMed 开源框架的医疗文本脱敏工具（USB-OpenMed）的研发过程、技术架构、测试验证及部署实践。项目采用纯规则引擎实现中文 PII（个人身份信息）检测，结合 macOS Vision 框架 OCR 技术，实现了 USB 即插即用的离线医疗数据脱敏解决方案。同时，本文档也记录了个人技术博客的搭建与部署全流程，展示了"AI+医疗"领域的技术实践路径。

## 一、项目背景

### 1.1 问题定义

医疗数据脱敏是临床研究、AI 训练和跨机构数据共享的前提条件。然而，现有脱敏方案存在以下痛点：

| 痛点 | 说明 |
|------|------|
| **依赖云端 API** | 大多数方案需要联网调用云服务，医院内网环境无法使用 |
| **重型 ML 模型** | 需要 GPU 或至少 2GB 的 PyTorch 环境，部署门槛高 |
| **需要管理员权限** | 医院电脑通常锁权限，pip install 无法完成 |
| **无 USB 便携方案** | 医生无法将工具带到不同科室/医院使用 |

### 1.2 解决方案

基于 OpenMed v1.5.5（Apache-2.0 开源，GitHub 3,540+ Stars）构建 **USB-OpenMed**——一套纯本地、零依赖、USB 即插即用的医疗文本脱敏工具。

**核心思路**: 放弃 ML 模型推理，采用纯规则引擎实现 PII 检测，从而消除 GPU 依赖和模型下载需求。

## 二、技术架构

### 2.1 系统分层

```

├─────────────────────────────────────────────────────────┤
│                  脱敏引擎层                              │
│   12 种 PII 正则模式 + 1,200+ 医疗关键词 + 5 种脱敏策略    │

│                  数据存储层                              │

```

### 2.2 PII 检测体系（12 种模式）

| 类别 | 检测内容 | 示例 |
|------|----------|------|
| **患者姓名** | 中文姓名、英文名、姓氏前缀 | "患者张三"、"娃名李四" |
| **医师信息** | 医师签名、科室、职称 | "张医生"、"李主任" |
| **联系方式** | 手机号、固话 | 138xxxx8888 |
| **证件号码** | 身份证、医保号、登记号、床号 | 440529199001011234 |
| **地址** | 省/市/区/街道/路/号 | "广州市天河区" |
| **日期** | 诊疗日期、出生日期 | "2024年3月15日" |
| **年龄** | 年龄数值 | "35岁" |

手工构建的中文医疗词典，涵盖 **1,200+ 关键词**，按 6 类组织：

| 类别 | 数量 | 示例 |
|------|------|------|
| 疾病/诊断 | ~400 | 高血压、糖尿病、骨折、肿瘤 |
| 药物/治疗 | ~300 | 阿司匹林、化疗、静脉注射 |
| 症状/体征 | ~200 | 发热、咳嗽、水肿、疼痛 |
| 解剖部位 | ~150 | 膝关节、腰椎、肺部、心脏 |
| 检查/化验 | ~100 | CT、MRI、血常规、尿常规 |
| 其他医疗术语 | ~50 | 过敏史、家族史、既往史 |

### 2.4 脱敏策略（5 种）

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **mask** | 用 `*` 替换 | 完全隐藏 |
| **remove** | 直接删除 | 彻底移除 |
| **replace** | 替换为伪名 | 保持文本结构 |
| **hash** | 哈希值 | 可追溯但不可逆 |
| **shift_dates** | 日期偏移 | 保持时间关系 |

## 三、关键挑战与解决方案

### 3.1 OCR 方案选型

**问题**: 初始选择 Tesseract，但在 macOS Monterey（x86_64）上遇到严重问题：
- Homebrew 被列为 Tier 3 支持
- 安装 Tesseract 时卡在 gnome.org 依赖下载

**解决方案**: 使用 Swift 编译 macOS 原生的 Vision 框架 OCR 工具

```bash
swiftc ocr_tool.swift -o ocr_tool
```

**优势**:
- 原生 OCR 在速度和中文识别率上均优于 Tesseract
- 无任何外部依赖
- 编译仅需一行命令

### 3.2 假阳性抑制

**问题**: 纯规则引擎面临的最大问题是假阳性（如"程度一"被误判为姓名）

**解决方案**: 黑名单机制

```python
BLACKLIST = ["程度一", "白蛋白", "第一", "第二", "类型一"]
```

### 3.3 别名归一化

**问题**: 医疗文档中同一实体有多种表述方式

**解决方案**: 别名映射层

```
"糖尿病患者" / "糖尿病病人" → "糖尿病"
"复方丹参滴丸" / "丹参滴丸" → "丹参滴丸"
```

### 3.4 一致性替代

**问题**: 同一患者的同一实体在不同文档中应被替换为相同的伪名

**解决方案**: 确定性伪名生成

```
基于 实体原文 + 命名空间 + 种子 → 哈希映射
保证同输入同输出
```

### 8.2 核心经验

1. **规则引擎在特定场景下比 ML 更可靠** — 确定性输出、零依赖、可解释
2. **本地优先（Local-first）是医疗 AI 的关键** — 数据不出院、无需联网、合规友好
3. **USB 便携化是落地关键** — 医生无需懂技术，插 U 盘即用
4. **技术博客是个人品牌的重要载体** — 记录思考、展示项目、建立影响力
